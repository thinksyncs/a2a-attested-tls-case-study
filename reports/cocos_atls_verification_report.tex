\begin{abstract}
Motivated by the recent IETF agent2agent discussion about how to distinguish
informal security goals from machine-checked claims, this report presents a
small machine-checked case study of Cocos AI aTLS~\cite{cocosrepo}. In this
workspace, the legacy/current comparison already shows that moving to the
post-handshake design does not by itself resolve same-endpoint authenticity
under the explicit leakage assumption used by the relay models. The current
verifier additionally exposes canonical exporter-label acceptance and a weaker
missing-attestation-after-offer issue. The workspace combines implementation
regression tests, Tamarin models, and ProVerif~\cite{tamarin,proverif} models,
and also includes narrower checks for leaf-key substitution and session-scoped
context reuse. It therefore distinguishes informal security goals for the
design from a smaller set of machine-checked claims that this workspace can
already support directly.
\end{abstract}

\section{Overview}

This report is framed as a machine-checked case study motivated by the recent
agent2agent discussion about protocol goals and machine-checked claims. Within
that framing, the workspace is centered on a small set of security questions
about attested origin, binding completeness, and same-endpoint authenticity.
Its main result is already visible at that level: moving from the legacy
pre-handshake design to the current post-handshake design does not by itself
resolve same-endpoint authenticity under the explicit leakage assumption used
by the relay models, while the current verifier additionally exposes
canonical exporter-label acceptance and a weaker missing-attestation-after-offer
issue.
For authenticated AI infrastructure, there is also a separate identity
question beyond channel ownership alone: an accepted peer may still need to be
distinguished from another co-resident AI agent or local stand-in on the same
attested machine.

Across the workspace, Go-language regressions act as implementation anchors for
concrete verifier behavior. Tamarin is used for request/offer gating,
attestation presence, canonical-label enforcement, replay/one-shot behavior,
and a same-endpoint attack trace under leakage. ProVerif is used for
correspondence-style clarification of attested origin vs.\ same-endpoint
authenticity, including the legacy/current comparison.

This workspace was also motivated by the IETF agent2agent mailing-list
discussion around how to separate informal security goals from smaller
machine-checked claims in protocol analysis~\cite{a2aml,fatt}.

The current workspace is organized around a small verification tree plus a
small set of implementation regressions.\footnote{Main repo locations:
\path{verification/} and \path{pkg/atls/ea/authenticator_test.go}.}

This report should not be read as a condemnation of Cocos AI or of the broader
goal of authenticated AI infrastructure. The project is attempting ambitious
work at the edge of current protocol and attestation practice, and that is
precisely why early machine-checked scrutiny is useful here. The intent of this
workspace is to make design assumptions and boundary conditions explicit while
the design is still evolving, not to dismiss the underlying direction.

\section{Scope}

After the legacy pre-handshake design anchored around commit
\texttt{e372cfc}, Cocos AI later introduced the current post-handshake
exported-authenticator form anchored around commit \texttt{80bf813} (as of 9
April 2026).

This report focuses on that current post-handshake aTLS behavior, while
keeping a compact legacy/current design split and a minimal reproducibility
path for the older pre-handshake design. On the current side, it examines
relay/diversion, canonical exporter-label enforcement, missing attestation after
offer, leaf-key substitution, and
context-reuse questions through a mix of implementation tests and small formal
models.

More concretely, it focuses on the exported-authenticator\footnote{In this
report, ``exported authenticator'' refers to the post-handshake TLS
authenticator exchange used by the current Cocos AI design, rather than a
certificate message carried inside the initial handshake~\cite{rfc9261}.} path used around
Cocos AI aTLS:

\begin{enumerate}
\item the client completes a TLS 1.3 handshake,
\item the client sends an exported-authenticator request with a fresh request
context,
\item the server constructs an authenticator and may attach CMW
attestation\footnote{In this report, ``CMW attestation'' refers to the
attestation payload and certificate-extension format used by the current Cocos
AI aTLS implementation.} data as a leaf extension,
\item the client validates the authenticator and then verifies the attestation
binder against the connection state, request context, and leaf public key.
\end{enumerate}

The modeling style is deliberately minimal. Each artifact aims to be a small
working model for one question at a time, rather than a single monolithic proof
attempt for the entire protocol stack.

\section{Assumptions}

\begin{itemize}
\item The relay/diversion analysis uses an explicit leakage assumption: the
attested ephemeral private key is revealed after attestation issuance.
\item The context-reuse result is session-scoped: with
\texttt{ea.Session}\footnote{In this report, \texttt{ea.Session} refers to the
verifier-side state used to remember which request contexts have already been
accepted, so that the same context cannot be accepted twice.} the
workspace checks a one-shot guarantee; without session tracking, replay remains
possible at the API level.
\end{itemize}

\section{Out of Scope}

\begin{itemize}
\item A full symbolic model of TLS 1.3 or of the complete Cocos AI
implementation.
\item End-to-end exploitability of the missing-attestation-after-offer result under the default
server wiring.
\item Full X.509 certificate-chain validation and complete evidence-format
parsing.
\end{itemize}

\section{Design Split: Legacy vs. Current}

Cocos AI aTLS should be read as two different designs rather than one continuous
protocol:

\begin{itemize}
\item \textbf{Legacy pre-handshake aTLS.}
The older design used OpenSSL custom extensions during the TLS 1.3 handshake.
In that design, an evidence request was sent in \texttt{ClientHello}, a TEE
type value was carried in \texttt{EncryptedExtensions}, and the attestation
report was attached as a certificate extension.
\item \textbf{Current post-handshake aTLS.}
The current design completes the TLS 1.3 handshake first and then performs a
post-handshake exported-authenticator exchange that carries attestation data in
the authenticator response.
\end{itemize}

This report focuses primarily on the current post-handshake design, but it also
keeps a minimal reproducibility path for the legacy pre-handshake design so
that the two generations are not conflated.

The immediate design lesson from comparing the two generations is narrow but
important. Moving attestation from the handshake to a post-handshake
exported-authenticator exchange changes message placement and some binding
inputs, but it does not by itself establish same-endpoint authenticity. On the
relay question modeled in this report,
the key requirement is not only where attestation is carried, but whether the
accepted peer is demonstrably tied to the client's exact live channel.

\section{Verification Artifacts and Implementation Flow}

The workspace is organized around three narrowly scoped security questions:

\begin{enumerate}
\item whether client acceptance preserves origin authenticity at the
attestation level,
\item what is explicitly bound into the attestation and binder inputs,
especially the request context, the leaf public key, and any TLS
exporter-derived values, and
\item whether client acceptance further establishes same-endpoint
authenticity, rather than only attested origin authenticity plus proof of
possession of the attested leaf key.
\end{enumerate}

The detailed code-path inventory and artifact layout are kept in the
verification README.\footnote{\path{verification/README.md}. For the current
design, the main moving parts are the internal transport request/response path,
the attestation provider, and the binder/verifier helpers under
\path{pkg/atls/}.}

\subsection{Current Post-Handshake Flow}

At the implementation level, the current Cocos AI flow relevant to this report is:

\begin{enumerate}
\item the client first completes a TLS 1.3 handshake,
\item the client builds an exported-authenticator request containing a fresh
context and the attestation offer extension,
\item the server receives that request and calls
\texttt{BuildLeafExtensions(...)} to construct the leaf certificate-entry
extensions for the authenticator response,
\item the default provider computes a binding from
\texttt{ExportKeyingMaterial(label, context)} and the leaf public key, then
embeds evidence plus binder values into a CMW attestation extension,
\item the client verifies the authenticator and, when the extension is present,
parses the attestation payload and verifies the binder.
\end{enumerate}

This is the execution path that the current verification artifacts try to
mirror.

\section{Current Design Results}

\subsection{Implementation Regression Tests (Go-language)}

The current-design Go regressions live in the main authenticator test
file.\footnote{\path{pkg/atls/ea/authenticator_test.go}} Their role here is
not to provide a formal security proof, but to act as implementation anchors
for several narrow security-relevant behaviors in the current code path. In
the report, the important point is what those tests check rather than their
exact names. The current set covers four behaviors:

\begin{itemize}
\item acceptance of a non-canonical exporter label during attestation
verification,
\item API-level acceptance when an explicit attestation offer is followed by an
authenticator without an attestation extension,
\item a narrow positive check against leaf-key substitution, and
\item the split between session-scoped one-shot context handling and replayable
API-level validation without session tracking.
\end{itemize}

The exact test names and reproduction commands are documented in the
verification README.\footnote{\path{verification/README.md}}

\section{Limits and Interpretation}

Taken together, the workspace narrows several important questions to
machine-checked results, but it does not elevate every design goal into a
proved theorem.

\subsection{Relay / same-endpoint authenticity under leakage}

The relay/diversion and same-endpoint authenticity results are design-level and depend on the explicit
ephemeral-key leakage assumption used by the ProVerif and Tamarin models. They
are best read as evidence aligned with the relay/diversion risk reported in
CVE-2026-33697~\cite{ghsa-cve,nvd-cve,osv-cve}, not as a full end-to-end exploit proof for the complete
implementation.

Within this workspace, the move from the legacy pre-handshake design to the
current post-handshake design does not eliminate the same-endpoint
authenticity failure under that explicit leakage assumption. In that narrow
CVE-aligned sense, the redesign by itself does not yet provide evidence that
the issue has been resolved.

For authenticated AI infrastructure, same-endpoint authenticity is not an
optional refinement: it is needed to distinguish the genuine remote endpoint
from a relay or stand-in that can present valid attestation material. That
requires more than a redesign of message placement alone. At minimum, the
protocol needs a proof that the accepted peer is bound to the exact live
channel seen by the client, rather than only to attested origin plus
possession of the attested key. Under the explicit leakage assumption used
here, that likely requires either a stronger non-exportability assumption for
the attested key material or an additional non-transferable channel-bound
proof that a relay attacker cannot replay from a different endpoint.
Even that would still leave a separate AI-agent identity question. In a
multi-agent machine, proving the exact live channel is not yet the same thing
as proving that the accepted peer is the intended agent rather than another
co-resident agent or local stand-in using the same attested machine.

\subsection{Canonical Exporter-Label Enforcement}

The canonical exporter-label issue is the strongest additional finding in this
workspace because it is supported by three separate anchors: a falsified
Tamarin property, a ProVerif correspondence whose canonical-label condition
fails, and a Go regression showing acceptance of an alternate exporter label.

\subsection{Missing Attestation After Offer}

The missing-attestation-after-offer issue is weaker. The workspace supports it
with a falsified Tamarin property and a Go regression, but the result remains
API-level. The default server wiring appears more fail-closed than this
verifier behavior alone suggests, so end-to-end exploitability is not
established here.

\subsection{Leaf-key substitution and context reuse}

The workspace also includes two narrower results. First, leaf-key substitution
is supported by a Go regression and by a ProVerif correspondence in which
\texttt{ClientAccepts ==> ServerAttestsLeafKey} holds. Second, request-context
one-shot behavior is established only for the explicit \texttt{ea.Session}
path; without session tracking, replay remains possible at the API level. In
that sense, the current design shows a scoped one-shot guarantee rather than a
general replay-resistant result.

\section{Open Problems}

This workspace narrows several questions to machine-checked results, but it
also leaves a short list of concrete follow-up checks:

\begin{enumerate}
\item how to guarantee same-endpoint authenticity even under the explicit
leakage assumption used by the relay models,
\item how to distinguish the intended remote AI agent from another
co-resident agent or local stand-in on the same attested machine,
\item whether canonical exporter-label enforcement can be made fail-closed in
the verifier,
\item whether missing attestation after an explicit offer is blocked
end-to-end in the default server wiring,
\item how much replay resistance can be guaranteed outside the explicit
\texttt{ea.Session} path,
\item and whether extension-placement / certificate-entry confusion should be
checked more broadly beyond the current path already covered here.
\end{enumerate}

\section{Conclusion}

Early machine-checked scrutiny is useful here because it already exposes
meaningful protocol results early in the design cycle. In particular, the
legacy/current comparison shows that relocating attestation into the
post-handshake exported-authenticator path does not by itself resolve
same-endpoint authenticity on the relay question modeled here. The current
verifier path also exposes additional problems beyond
that published relay-aligned concern, most clearly in canonical
exporter-label enforcement and more weakly in missing attestation after
offer. More broadly, attested machine identity is not yet the same thing as
intended AI-agent identity on that machine. Together with the narrower
leaf-key and session-scoped replay results, this makes the workspace a scoped,
reproducible starting point for further checking rather than a complete
endpoint-verification story.
